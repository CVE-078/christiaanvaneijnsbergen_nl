# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

Package manager is **bun** (`bun.lock` is the source of truth; a `package-lock.json` also exists). Scripts run the same under `npm run`.

- `bun run dev`, start the Next.js dev server (http://localhost:3000)
- `bun run build`, production build
- `bun run lint`, ESLint (`next/core-web-vitals`)
- `bun run typecheck`, `tsc --noEmit`
- `bun run format` / `format:check`, Prettier over `src/**/*.{ts,tsx}`
- `bun run test`, Vitest in watch mode
- `bun run test:run`, Vitest single run (use this in CI / verification)
- `bun run test:ui`, Vitest UI

Run a single test file: `bun run test:run src/components/pulse/__tests__/SetLogger.test.tsx`
Run tests matching a name: `bun run test:run -t "warmup"`

## Tech Stack

Next.js 15 (App Router) · React 19 · TypeScript (strict) · Tailwind CSS v4 · Supabase (auth + Postgres) · SWR · Vitest + Testing Library (jsdom).

Path alias: `@/*` → `src/*` (configured in both `tsconfig.json` and `vitest.config.mjs`).

2. **Pulse** (`/pulse`), a workout-tracking, installable PWA where nearly all the complexity lives. Dark "Slate" theme; Hanken Grotesk + Sora + Big Shoulders Display (loaded via `next/font` in `PulseLayout`). Everything Pulse-related is namespaced under `pulse/` directories.

## Pulse architecture

The big picture (read `docs/superpowers/specs/2026-05-25-pulse-architecture-design.md` for the original design):

**Server → Context → Hooks → Components**, with SWR keeping client data live.

- **Auth & data load**, `src/middleware.ts` (matcher `/pulse/:path*`) calls `updateSession` in `src/lib/supabase/middleware.ts`, which refreshes the Supabase session and redirects unauthenticated users to `/pulse/login`. Routes live under the `(protected)` route group; `src/app/pulse/(protected)/layout.tsx` is a thin server component that only resolves the user (redirect if absent) and renders the shell, it does **not** prefetch data. Data is fetched **client-side** by the SWR hooks (shell-first / instant loading), so the app shell paints immediately and each view shows a skeleton until its slice arrives.
- **Three Supabase clients**, `src/lib/supabase/server.ts` (server components / actions, async, cookie-based), `browser.ts` (client), `middleware.ts` (edge session refresh). Use the right one for the context.
- **PulseLayout → PulseProvider → AppShell**, `PulseLayout` maps the URL pathname to a `View` and supplies `navigate`. `PulseProvider` (`src/components/pulse/PulseProvider.tsx`) composes the hooks and exposes everything through `PulseContext`. Consume state anywhere with `usePulse()` from `src/context/PulseContext.ts`, that file's `PulseContextValue` interface is the canonical contract for what the app can do.
- **Hooks** (`src/hooks/pulse/`), one hook per data domain: `useWorkoutLogs`, `useProfile`, `useRoutines`, `useNotes`, `useWorkoutSession`, `useDecisionEvents` (read-only feed of the adaptive `DecisionEvent` log, surfaced by the Coach Decision Timeline / `CoachPanel`), `useEquipmentProfiles` (named reusable equipment sets, see "Equipment profiles" below), `useProgramPauses` (program pause / injury mode; optimistic pause/resume over the `program_pauses` table, see "Periodized program" below), plus UI hooks `useUIState`, `useRestTimer`, `useMediaQuery`, `useLocalStorage`. Each data hook follows the same pattern: `useSWR` keyed on its `/api/pulse/*` endpoint (fetches on mount; expose `loading`/`error`, default to a stable empty value), then **optimistic mutations**, call `mutate(newValue, false)` immediately, await the server action, then `mutate()` to revalidate. A user-scoped `localStorage` SWR cache (`src/lib/pulse/swrCache.ts`, wired in `PulseLayout`, cleared on logout) makes warm visits instant via stale-while-revalidate.
- **Server actions vs API routes**, Mutations go through `'use server'` actions, domain-split under `src/app/pulse/actions/*` (`session.ts`, `routines.ts`, `swaps.ts`, `equipment.ts`, `pause.ts`, `adjustments.ts`, `decisions.ts`, `notes.ts`, `exercises.ts`, `profile.ts`) and re-exported through the `src/app/pulse/actions.ts` barrel (plus `login/actions.ts`); set logs use the per-row `upsertLog`/`deleteLogRow` (in `actions/session.ts`), not a bulk save. Reads (and SWR revalidation) go through GET handlers in `src/app/api/pulse/*/route.ts`, which reuse the loaders in `src/lib/pulse/queries.ts`. When you add a data domain you typically touch: the action, the API route, the hook, and the context interface. Set-log/note mutations also pass through an **IndexedDB offline write queue** (`offlineQueue.ts` / `offlineSync.ts`): writes enqueue when offline or on failure and flush FIFO on reconnect/focus, so the optimistic UI stays live offline (Pulse is an installable PWA with a `/pulse`-scoped service worker).
- **Responsive split**, `AppShell` branches on `useMediaQuery('(min-width: 1024px)')`: `DesktopLayout` (sidebar) for desktop, `BottomNav` + topbar for mobile. The same view components render in both.
- **Views** (`src/components/pulse/views/`), `LogView` (train), `ProgramView` (plan), `HistoryView` + progress charts, `ProfileView`, `LibraryView` (+ `TemplatesTab`). Routed via `src/app/pulse/(protected)/{train,plan,progress,profile,library}`.
- **Guided mode**, `WorkoutModeScreen` (full-screen overlay opened from `LogView`) branches on `useMediaQuery` like the shell: a single-column "Focus" layout on mobile, a two-pane layout (focus pane + session sidebar) on desktop. `SetLogger`, the per-set logger shared with the Train `ExerciseCard`, takes `variant: 'card' | 'editorial'`, `card` is the Train look, `editorial` the guided look; default `card` so Train is unaffected. The `editorial` input state carries the set-row polish: prescription chips (rep target + RIR), "Same as last" (fills from `previousEntry`), `totalSets` for "Set N / total", and a per-set ⋯ overflow (Add drop set). The plate calculator shows only when `plateLoaded` (callers derive it via `isPlateLoaded`, barbell-only) and closes on save; `PlateCalculator` renders a loaded-bar visual.

## Domain model (Pulse)

- **Types**, `src/lib/pulse/types.ts` is the single source for all Pulse types.
- **Pure logic**, `src/lib/pulse/utils.ts` holds the testable pure functions: `computeStreak`, `computePRMap`, `computeLastSession`, `computeWarmupSets`, `computeShareStats`, the program helpers `getPhase` / `getRIR` / `volumeForWeek` / `computeProgression`, and the plateau/deload trio `computePlateau` / `shouldDeload` / `deloadTarget`. Prefer adding pure functions here (easy to unit-test) over logic in components. Other lib modules: `data.ts` (`buildProgram(weeks)`, the periodized block tables), `adherence.ts` (calendar/completion-paced engine), `generation.ts` (routine generation), `recommendation.ts` (onboarding template match), `strength.ts`, `offlineQueue.ts` / `offlineSync.ts` (offline write queue), `templateFilters.ts`, `csv.ts` (history export), `historyBundle.ts`, plus `sessions.ts` (holds `nextVariant`), `validation.ts`, `constants.ts`.
- **Log key format**, set logs are keyed `"week-routineExerciseId-setIdx"` where the middle segment is a UUID (contains dashes). Parse with `indexOf('-')` / `lastIndexOf('-')`, not `split` (see `actions.ts`). Notes are keyed `"week-routineExerciseId"`.
- **Periodized program**, a routine carries `program_weeks` (8/10/12/16, default 12). `buildProgram(weeks)` in `data.ts` defines one repeating **block** (12 = the legacy tables); `getPhase` / `getRIR` / `volumeForWeek` are block-length-aware and **wrap** (`weekInBlock`) so the program repeats indefinitely and deloads at each block end. It is also **completion-paced and calendar-aware**: a routine's `program_anchor` + the user's `timezone` let `adherence.ts` attribute completed sessions to program weeks, slide the timeline forward on missed sessions, and surface a non-destructive **ramp-back** nudge after a gap, plus a user-initiated **lighten this week** (`manual_deload`) that eases the current week without inserting one (decisions persist in the append-only `program_adjustments` table; ramp-back and manual lightens also dual-write to the `decision_events` log read by the Coach Decision Timeline). A user-initiated **program pause / injury mode** (Tier 3 #14) freezes program time while active: it lives in its own `program_pauses` table (a date span: `paused_at`, nullable `resumed_at`, `resumed_at IS NULL` = active, a partial unique index allows one active pause per routine; it is NOT an `AdjustmentKind`, a pause spans many weeks and would collide with the week-keyed `program_adjustments` unique constraint). While paused, status reads `paused` (a new `AdherenceStatus`), `behindBy` excludes scheduled sessions that fell in a paused span, `calendarWeek` subtracts paused days, and the regen nudge is suppressed; `daysSinceLastSession` stays the *real* gap (never pause-adjusted) so resuming from a long pause still hands off to the existing ramp-back. `adherence.ts` carries `activePause` + interval-union helpers threaded into `computeProgramPosition` / `computeWeekAdherence` / `computeRegenSuggestion` (`progressionInfo` is untouched, no calendar input); `pauseProgram` / `resumeProgram` live in `actions/pause.ts` (23505-idempotent, ownership-checked), with `useProgramPauses` + a Train control. No `decision_events` dual-write in v1 (a date-based pause does not dedupe in the week-keyed log; Coach Timeline pause entries + a `reason` picker are the v2 follow-on). Stalled main lifts auto-deload: `shouldDeload` / `deloadTarget` prefill ~90% of the last weight in `SetLogger`. Workout types: `push/pull/legs` are legacy; granular types (`chest/back/shoulders/arms`) fall back to their parent.
- **A–D variants**, `WorkoutVariant` is `'A' | 'B' | 'C' | 'D'` (widened from A/B so multi-session full-body styles fit). `routine_exercises` and `routine_schedule` both carry a `variant` column; `routine_schedule.variant` **pins each scheduled day to a session** (e.g. Monday = Upper A), so day-select jumps straight to that tab in `PulseProvider`. `WorkoutTabs` sorts/labels A→D by string compare (no per-letter logic); `routineExercisesByTabKey` is computed in the provider.
- **Routine generation**, `src/lib/pulse/generation.ts` builds routines from a **program style** (a named `(focus, emphasis, variant)` sequence per training day, keyed by session count in `STYLES`). Each session uses an **emphasis profile** (`EMPHASES`: a `bias` + ordered movement-pattern slots); repeated focuses (two Upper days, three Full-Body days) get different emphases so they vary by design, not by accident. Paired leg days are quad-led vs posterior-led by composition (`lower_quad` / `lower_lean` carry `squat` not `hinge`, `lower_post` carries `hinge` not `squat`), so the `consistent` anchor cannot pin the same main lifts to both; `vertical_pull` lives only in the back-focused upper/pull emphases (`pull`, `upper_general`, `upper_chest_back`, `upper_aesthetic_a`) and no-ops for dumbbell-only users (the equipment filter skips the slot, backfill covers it). `push` / `pull` carry a deliberate 6th slot (a 2nd `triceps_iso` / `back_iso`). The shared engine filters the pool by selected equipment, fills slots with a routine-wide **avoid-set** (cross-session variation, no within-session dupes), sets rep ranges via `repRange(bias, isCompound, goal)`, and for `~30 min` sessions pairs antagonist exercises into **auto-supersets**. Within a session, `byPattern` ranks same-pattern candidates through a layered sort: loading-lean equipment match, substitution-class freshness (`usedSubstitutionClasses`, routine-wide, soft-deprioritizes a repeat of an already-seated `substitution_class`, e.g. a second Romanian Deadlift variant on Legs after Pull already used one), front-delt-isolation suppression (the `front_delt_isolation` substitution class is soft-deprioritized once a vertical press fills the session, pressing already loads the front delts), a **canonical-anchor rank** (`CANONICAL_ANCHORS`, a name-keyed map of the standard primary compound per anchor pattern) applied **BEFORE the fatigue tiebreak** (changed 2026-06-10): for explicitly named anchors the canonical preference is authoritative, so e.g. **Romanian Deadlift anchors `hinge`** over the higher-fatigue Deadlift / Sumo Deadlift (both fatigue 5), and Barbell Bench Press stays ahead of Close-Grip Bench Press. Listed patterns: horizontal_push / vertical_push / horizontal_pull / vertical_pull / hinge (RDL > Deadlift > Sumo); squat is NOT listed (Barbell Squat alone at fatigue 5 resolves cleanly). It still sits below loading-lean + substitution-class freshness, and is a no-op for nameless / unlisted exercises (`Infinity` rank), so synthetic pools stay byte-identical; fed by `name` on `ExerciseMeta`, threaded through `actions/routines.ts`; keyed by name because ids are UUIDs, guarded by a catalog-consistency test, longer-term an `anchor_rank` column. Then a role-aware **fatigue tiebreak** (`fatigue?: number`, neutral midpoint 3 when absent; the six `COMPOUND_ANCHOR_PATTERNS` -- squat / hinge / horizontal_push / vertical_push / horizontal_pull / vertical_pull -- prefer the *higher*-fatigue, better anchor when fresh, e.g. Bench Press over Push-Up, while accessory patterns prefer the lower-fatigue option), now a general tiebreak below canonical rather than above it (accepted side effect: the 2nd vertical-push pick is Dumbbell OHP over Dumbbell Push Press, the 2nd vertical-pull pick Lat Pulldown over Chin-Up, both test-locked). Then alphabetical id. A post-select **tier sort** orders compounds before isolation, then a **squat/hinge interleave** (`interleaveLowerCompounds`, 2026-06-10) slots the first upper compound between squat and hinge when a session leads with both (so the two highest-fatigue lifts don't open back to back; leg-only sessions with no upper compound, and position 0 the set-bump target, are untouched). Three hard caps gate `pick`: a **max-2-per-movement-pattern cap** (the diversity rail, NEVER relaxed -- a session under-fills toward variety rather than stacking a 3rd of one pattern, e.g. three calf raises; the deliberate push/pull 2nd triceps_iso/back_iso sits exactly at the cap), `HEAVY_DEDUP_PATTERNS` (hinge / squat) blocks a second heavy compound per session, and the `unilateral` cap (`ExerciseMeta.unilateral: boolean`) blocks a second single-limb **compound** lift per session regardless of pattern (Walking Lunge cannot also seat Bulgarian Split Squat); the cap is scoped by `unilateralCapApplies(pattern)` so isolation / finisher slots (`_iso` / calf / core) are EXEMPT (a unilateral Cable Kickback neither consumes the budget nor is blocked by it -- without this, a fresh unilateral glute accessory starved the `lower_post` lunge slot, the real cause of the "Legs missing a lunge / Abduction in both" symptom, fixed 2026-06-10); the latter two relax only after a full backfill round yields nothing (`relaxedHeavyCap` / `relaxedUnilateralCap`), so thin equipment pools still reach the target count, and backfill prefers the **least-represented pattern** (diversity scoring, breadth before depth). `recommendStyle(count)` picks the default style; `RoutineSetupFlow` shows a style-picker step when a count has more than one style. **Training style** (`TrainingStyle` = Balanced / Strength / Bodybuilding / Powerbuilding) is a second axis over `goal`, chosen in an optional `RoutineSetupFlow` step and persisted on `profiles.training_style` (nullable, mirrors `priority_muscle`): `resolveBias(sessionBias, style)` remaps each session's bias through a 4×4 table (Balanced is the identity, so existing routines regenerate byte-identical), and `resolveRepRange` defers to `repRange` except for Powerbuilding, which reserves the strength range for `POWERBUILDING_HEAVY_PATTERNS` (squat / hinge / horizontal_push / vertical_push / horizontal_pull / vertical_pull, the two pulls added 2026-06-10 so a powerbuilding pull day trains heavy instead of getting hypertrophy reps while the first compound still took the strength set-bump) and gives accessories hypertrophy. The generate action takes `trainingStyle` (param wins over the stored profile value, then written back); `buildRationale` appends a style clause. Training style does NOT constrain frequency (no fatigue model yet). **Variety preference** (`VarietyPreference` = 'consistent' | 'varied') is a third, fully orthogonal axis that controls *which* exercise fills a slot (not how it is trained): under `consistent`, a per-generation, never-persisted anchor map (`Map<string, exerciseId>` keyed by `` `${focus}:${pattern}` ``) in `selectForSession` pins the main compound lifts (`COMPOUND_ANCHOR_PATTERNS` = squat / hinge / horizontal_push / vertical_push / horizontal_pull / vertical_pull, a SEPARATE constant from `POWERBUILDING_HEAVY_PATTERNS`) across sessions of the **same focus** while accessories keep the fresh-preference; the anchor lookup takes precedence over the avoid-set for those slots. Keying by `(focus, pattern)` (not pattern alone) is deliberate: two same-focus days (Lower A + Lower B) share an anchor, but different-focus days (Push vs Upper, Legs vs Lower) get distinct keys so the second one's fresh pick respects the avoid-set and they never collapse to the same exercise (the cross-session collapse bug). The key relies on `Focus` / `MovementPattern` enum members being colon-free. `varied` (default, `input.varietyPreference ?? 'varied'`, also the null/undefined case) is the byte-identical no-op path, locked by a golden identity + `consistent`-determinism test. Persisted on `profiles.variety_preference` (nullable, mirrors `training_style`), chosen in an optional `RoutineSetupFlow` step (`collectVariety`), threaded into the generate action exactly like `trainingStyle`. **Loading lean** (`LoadingPreference` = 'barbell' / 'dumbbell' / 'machine' / 'cable') is a fourth axis: it floats preferred-equipment exercises first in `byPattern` (fresh-preference still wins over a used preferred), persisted on `profiles.loading_lean` (nullable), an optional `RoutineSetupFlow` 'loading' step (toggle-to-deselect, "Skip" when nothing chosen); null/undefined is byte-identical to base, `TemplatesTab` opts out (`collectLoadingLean={false}`). **Movement restrictions** (`RestrictionFlag` = 'knee' / 'lower_back' / 'shoulder' / 'wrist') are a fifth axis, but a HARD **pool filter** not a sort: `isContraindicated(ex, restrictions)` runs beside `hasEquipment` so a flagged joint excludes every exercise carrying that flag in its per-exercise `contraindications: RestrictionFlag[]` metadata. It is never relaxed (a contraindicated lift is never re-added to fill a slot, unlike the equipment / heavy-dedup / unilateral relax fallbacks); an emptied pattern is covered by the existing backfill from safe patterns. A **minimum-compound guard** (Item 2, 2026-06-10) backstops the subtractive filter: a session whose compound patterns were all filtered out (e.g. knee + lower_back emptying squat / hinge / lunge) seats one safe compound from any other pattern (cross-pattern fallback, dropping the lowest-priority isolation to hold the count) rather than shipping accessory-only; if no compound survives anywhere in the pool it appends a non-blocking notice (`RoutineBlueprint.warnings`) to the routine rationale and still generates. Intent-matched substitution (a safe posterior compound for an emptied hinge) is the v1.5 follow-on, not this guard. Persisted on `profiles.movement_restrictions` (nullable `text[]`), collected in an optional `RoutineSetupFlow` multi-select step (`collectRestrictions`, `TemplatesTab` opts out) AND a standing Profile editor (`updateMovementRestrictions`); empty/undefined is byte-identical to base. The generate action threads it with a **safety-aware write-back**: restrictions persist only when the param is explicitly passed (an absent param never clears a stored flag; an explicit `[]` does). v1 is purely subtractive (safe lifts surface because they remain, no preferred-substitute front-loading yet; future hook is `substitution_class`). Migrations `…-movement-restrictions-profile.sql` + `…-exercise-contraindications.sql` (the latter tags exercises against the corrected catalog with a safe-leg/safe-push survival invariant per flag). Standalone **Profile-screen editors** for training style / variety / loading lean now exist (a "Training preferences" group in `ProfileView`, alongside the restrictions editor), persist-only via `updateTrainingStyle` / `updateVarietyPreference` / `updateLoadingLean` (the latter two new; `updateTrainingStyle` was dormant and is now exposed through context), applying to the next generation. The loading-lean editor has an explicit "No preference" row (null is intentional, not a default) and its equipment rows do not tap-to-deselect (clear via "No preference"). All four editors' option label/desc constants live in the shared `src/lib/pulse/generationPreferences.ts` (imported by both `RoutineSetupFlow` and `ProfileView`). **Quick-start generation** (Tier 1 #15) then shipped: a `mode:'quick'` on `RoutineSetupFlow` trims first-generation to **6 steps** (equipment, experience, goal, days/week, session length, confirm+start), auto-applying `SUGGESTED_DAYS` + `recommendStyle` and dropping gender plus the four personalization steps (their inputs live in Profile / the panel below; "Change split" stays one tap away). The freshly generated routine then routes into `TuneYourPlanPanel`, a full-screen overlay offering "Change split" (only when the day count has more than one style) plus the four personalization pickers seeded from the stored profile; "Apply changes" stays disabled until a picker actually differs from what the routine was built with (a `dirty` check against an `applied` snapshot, so an identical re-run can't churn out a near-duplicate), then **regenerates in place**: generate fresh with the new params (auto-activating the new routine), carry over `program_anchor` / `program_weeks`, delete the old routine. Both `OnboardingModal` and `GenerateRoutineButton` hand off to the panel through a `handingOffRef` guard on the `onClose` they pass to `RoutineSetupFlow`, because `handleComplete`'s `finally` always calls `onClose` immediately after `onComplete` resolves and would otherwise unmount the panel mid-handoff (`OnboardingModal` additionally pins `showOnboarding` open via `triggerOnboarding()`, since `generateRoutine`'s SWR revalidation flips `routines.length` from 0 and would compute it closed). Its optional gender step (onboarding only) has a first-class "Prefer not to say" choice; no gender is valid and falls back to neutral defaults. - **Travel mode** (Tier 2 #322, **shipped 2026-06-09**, `feature/travel-mode`), a temporary equipment-profile **overlay** that auto-reverts. A nullable `expires_at` on `equipment_profiles` marks the travel set; while it is a FUTURE calendar day in the user's `timezone` that profile is the *effective* equipment set, past it the overlay is ignored **at read time** (no cron) and the effective set falls back to the untouched default (`active_equipment_profile_id`). Calendar math reuses the DST-safe `dayIndex` (moved into a cycle-free `src/lib/pulse/dates.ts`, re-exported from `adherence.ts` so `utils.ts` can import it without the `adherence -> utils` cycle); `expires_at` stores **noon-UTC of the return day** so the tz calendar day is stable, and `isTravelActive` treats the return day itself (`==`) as inactive. **One overlay per user** is enforced by a partial unique index `equipment_profiles_one_overlay_per_user (user_id) where expires_at is not null` plus an atomic clear-others-then-set in the `startTravel` action (the Supabase JS client can't express a per-row CASE; on a partial-set failure the user simply has no overlay, which is safe, and the hook's revalidate-in-`finally` rolls the optimistic patch back). ## Copy & writing

**No em dashes anywhere in the application.** Not in user-facing copy, not in code comments, not in new docs. Rework the sentence with a comma, period, semicolon, or parentheses, or use a plain hyphen where a dash truly reads best (template-name style separators use a hyphen, e.g. "Full Body - Dumbbells"). This applies to every string a user can see and to anything you author in the repo. The lone `—` glyph used as an empty-value placeholder (a "no data" cell) is the one exception and stays. The live `src/**`, `CLAUDE.md`, `docs/roadmap.md`, and the seeded DB content were swept clean on 2026-06-06; the historical `docs/superpowers/**` and `docs/audits/**` archives were left as point-in-time records, so if you open one, fix em dashes only in lines you actually touch.

## Styling

Tailwind v4 with theme tokens defined inline via `@theme` in `src/app/globals.css` (no `tailwind.config`). Pulse uses custom tokens: colors `pulse-bg`, `pulse-surface`, `pulse-surface-2`, `pulse-border`, `pulse-accent`, `pulse-text`, `pulse-dim`, `pulse-muted`, etc., and fonts `font-pulse` (Hanken Grotesk), `font-pulse-body` (Sora), `font-pulse-display` (Big Shoulders Display, big numerals/labels). Use these tokens rather than hardcoded hex. ## Security headers

`next.config.mjs` sets the static global security headers (X-Content-Type-Options, X-Frame-Options, Referrer-Policy, Permissions-Policy, HSTS) but deliberately does **not** set the CSP. The strict `/pulse` CSP is built **per-request in `src/lib/supabase/middleware.ts` (`buildCsp`, with a nonce + `strict-dynamic`)** and whitelists the Supabase host for `connect-src`/`wss`. If you add an external origin (CDN, API, font host) the Pulse pages talk to, update `buildCsp` in the middleware (not `next.config.mjs`) or requests will be blocked.

SQL migrations are hand-written and tracked in `docs/migrations/*.sql` (dated). There is no automated migration runner in this repo, apply them against Supabase manually. `scripts/migrate-kv-to-supabase.ts` is a one-off data backfill.

**Roadmap workflow (follow on every implementation; commit at each end):**

- **Start.** On the feature branch, mark the item *In progress* in `docs/roadmap.md`: set the `In progress:` line in the "Status & next session" block to the item name and the branch. - **Finish.** Move the item to **Shipped** (relocate or strike its tier-table row, add a dated Shipped bullet under "Reference & archive"), clear the `In progress:` line back to `(none)`, and bring `docs/roadmap.md` and this `CLAUDE.md` current with the codebase: the Status block (where things stand, next-up order), the test count, and any architecture/section the change touched. Commit the sync on the same branch as the final step. Keeping both docs accurate is part of "done", not optional.

**Roadmap hygiene (apply on every `docs/roadmap.md` edit, not only at Finish):** the roadmap is a living source of truth, not a brainstorm dump. Do this without being asked.

- **Separate by status.** Shipped, in-progress, deferred, and rejected each read clearly. When an item's status changes, update its existing entry, never add a parallel one. Scope each item to one decision or one deliverable.
- **Compress, don't narrate.** Preserve important decision history as short decision records, not long narrative blocks. Avoid repeating the same detail across sections. When in doubt: shorten, consolidate, clarify.
- **Remove what's resolved.** Delete duplicate ideas, stale notes, resolved questions, and old validation / temporary-triage comments once they no longer serve. If the roadmap states implementation details that are no longer true, rewrite or remove them.
- **Keep "next up" tight.** Limit it to the smallest useful set of actionable items.
- **Branches.** When a feature is split, label Branch A / Branch B and state the dependency between them.
- **Flag, don't bury.** Call out contradictions, outdated assumptions, and missing decisions explicitly. Prefer concrete names, dates, and status labels over vague language.
- **Keep the lanes separate.** Marketing copy, product vision, and execution plan live in their own sections.
- **Editing discipline.** Keep headings stable. Use precise, compact language (the no-em-dash rule applies). Do not expand scope while cleaning.
