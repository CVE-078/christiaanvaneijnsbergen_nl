# Phase 0 #1 — `muscleMap.ts` (MovementPattern → muscle bridge): implementation spec

**Status:** implemented 2026-06-06 on branch `feature/muscle-map`. Pure module, no migration. TDD (golden tests first).

## Goal

Build the canonical, weighted bridge from Pulse's 15 `MovementPattern`s to its 10 `ExerciseCategory`s as a pure, well-tested module. This is generation-v2 **Phase 0 #1** — the foundation the 2026-06-06 audit identified as the single missing piece: generation is slot-first over 15 movement patterns, analytics report over 10 categories, and the two vocabularies have no bridge. Everything later (volume-aware analytics, volume-first generation) depends on this map existing.

**Source of truth:** the frozen §1 map in `docs/superpowers/designs/2026-06-06-00-54-52-phase0-source-material.md`, under **granularity decision A** (keep the 10-category taxonomy, collapse only the muscle-target side; the 15 patterns remain the generation control layer — the three-layer rule from the roadmap's Product decisions). Weights transcribed verbatim, not re-derived.

## In scope (this task)

`src/lib/pulse/muscleMap.ts`:
- `type MuscleContribution = Partial<Record<ExerciseCategory, number>>`
- `PATTERN_MUSCLE_MAP: Record<MovementPattern, MuscleContribution>` — the frozen v1 weights (sum to 1.0 per pattern). `Record<MovementPattern, …>` gives a compile-time guarantee that all 15 patterns are present.
- `muscleContributions(pattern)` — the weighted contributions (returns a copy so the shared map can't be mutated).
- `primaryMuscle(pattern)` — the dominant category, ties resolved primary-first.

Golden tests (`__tests__/muscleMap.test.ts`): all 15 patterns covered; weights sum to 1.0; only the 10 real categories used; the exact frozen weights (locks the map against silent drift); the squat-vs-hinge glute-proportion invariant; accessor copy/dominant behavior.

## Out of scope — deferred, needs a decision before it ships

- **Wiring the bridge into `computePerMuscleVolume` (the volume-bar bugfix).** Today the Progress volume bars credit a full 1.0 to one category; the bridge fixes that. But the design doc leaves an **open decision**: **proportional** (weights sum to 1.0, a bench set = 0.55 chest) vs **fractional-set** (primary ≈ 1.0, secondaries fractional, per-set totals exceed 1.0 — compares better against the MEV/MAV/MRV landmarks). This changes the visible numbers on Progress and must stay consistent with how `VOLUME_TARGETS` is tuned. **Left for Christiaan to decide before wiring** (it's a visible behavior change, not pure infra).
- Exercise-metadata expansion (Phase 0 #2 — needs the ChatGPT step).
- `session_id` / workout date / `DecisionEvent` log (Phase 0 #3 — needs a hand-applied migration).
- Volume-first generation (Phase 4) — when it arrives it reads targets *through* this bridge, never the analytics layer directly.

## Why a module, not inline

Keeps the bridge a single data table so more patterns or finer muscle precision (the deferred quad/ham + delt split) can be added later without a rewrite, and keeps the control layer (patterns) and reporting layer (categories) bridged in exactly one place.
